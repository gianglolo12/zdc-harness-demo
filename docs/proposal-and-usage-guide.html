ZDC Harness — Đề xuất & Hướng dẫn vận hành

Hệ thống tự động hoá vòng đời "PRD → phân tích tác động → duyệt → code → PR" cho backend (và mở rộng cho frontend). Server host Claude Code, lắng nghe git webhook, chạy 2 pha có cổng duyệt của con người.

Provider: GitHub Deploy: Dokploy VPS Auth: Claude subscription ✓ Đã chạy thật end-to-end (PR #4)

1. Vấn đề & Giải pháp

Vấn đề

Mỗi khi PO cập nhật PRD, dev phải tự đọc, đánh giá tác động lên code, rồi code tay. Tốn thời gian, dễ sót, không đồng nhất giữa các dev.

Giải pháp

Một harness đứng giữa Git và Claude Code: PRD đổi → tự phân tích tác động và mở PR nháp → dev chỉ duyệt → harness tự code + test + đẩy PR hoàn chỉnh. Dev giữ quyền quyết định ở "cổng người".

Kiến trúc gọn

Git webhook→ server (Fastify)→ Redis queue→ worker→ claude -p (trong checkout)→ PR trên GitHub
Thành phầnVai trò
serverNhận POST /webhook, xác thực HMAC, phân loại sự kiện, đẩy job vào queue.
workerLấy job, clone repo, chạy claude -p các lệnh /auto-*, tạo/un-draft PR.
redisHàng đợi job (BullMQ) + lưu trạng thái MR (job:<mrIid>).
agent bundleThư mục examples/control-plane/: registry.yaml (map target→repo) + các lệnh .claude/commands/auto-*.md. Bake trong image, overlay lên checkout lúc chạy.
memorySQLite FTS5 lưu bài học/issue đã gặp, nạp lại cho các lần phân tích sau.

2. Chuẩn bị (làm 1 lần)

#ViệcChi tiết
1Agent bundleĐặt registry.yaml + .claude/commands/auto-{impact,review-solution,implement}.md trong examples/control-plane/. Team tự chỉnh prompt cho đúng nghiệp vụ.
2Biến môi trường (Dokploy → Environment)GIT_PROVIDER=github, GITHUB_OWNER, GITHUB_REPO, GITHUB_TOKEN (scope repo), WEBHOOK_SECRET, DRY_RUN=0, IS_SANDBOX=1.
3Đăng nhập ClaudeTrong container worker (Open Terminal): chạy claude → /login (gói subscription). Credential lưu ở volume, không cần API key.
4Webhook GitHubTrên repo đích → Settings → Webhooks: URL http://<domain>/webhook, content-type application/json, secret = WEBHOOK_SECRET, events push + issue_comment.
Lưu ý: IS_SANDBOX=1 bắt buộc — container chạy bằng root, Claude từ chối --dangerously-skip-permissions dưới root trừ khi biến này được đặt. Thiếu nó, cả 2 pha sẽ "im lặng không làm gì".

3. Các step khi chạy + hướng dẫn dùng

Pha 1 — Phân tích tác động → PR nháp

Tự động hoàn toàn. Đầu ra: 1 Pull Request nháp chứa bản phân tích tác động.

1
Trigger — push PRD · PO
Hệ thống: GitHub bắn webhook push tới server.
Bạn làm: trên một feature branch (không phải main), commit thay đổi file dưới po/**, với commit message chứa tag. 
# nhánh ví dụ: feature/zdc-be-PRD-005
git commit -m "docs(po): add PRD-005 list orders [zdc:update-be PRD-005]"
git push origin feature/zdc-be-PRD-005
Cú pháp tag: [zdc:update-<be|fe|qa> <PRD-ID>].
2
Xác thực + phân loại · server (tự động)
Kiểm HMAC X-Hub-Signature-256. Nếu branch hợp lệ + có tag + có file po/ → phân loại impact; ngược lại bỏ qua (200).
Bạn làm: không cần gì. Sai chữ ký → 401; thiếu tag → server trả {"type":"ignore"}.
3-5
Enqueue → worker clone + overlay · hệ thống
Job vào Redis. Worker clone repo (--depth=1 --branch <ref>) vào /tmp/zdc-checkout-* trên VPS, dán agent bundle lên trên, gắn token để push được ở Pha 2.
6-7
/auto-impact + second-opinion · Claude (tự động)
Nạp memory → claude -p /auto-impact đọc PRD + codebase → bản phân tích tác động. Sau đó /auto-review-solution chấm điểm; fail thì làm lại (≤2 vòng).
8
Tạo PR nháp + lưu trạng thái · hệ thống
Mở Pull Request nháp (draft) chứa phân tích; lưu state vào Redis. Dừng, chờ người duyệt.
Bạn làm: mở PR nháp vừa hiện trên GitHub, đọc phần "Impact analysis".
Cổng người — duyệt

Điểm con người ra quyết định. Tương tác bằng comment trên PR.

9
Đọc & quyết định · BE Dev
Bạn làm: comment lên PR một trong các lệnh:
LệnhÝ nghĩa
/approveĐồng ý phân tích → mở Pha 2 (code thật).
/revise <góp ý>Yêu cầu phân tích lại kèm góp ý. Tối đa 3 lần, sau đó gắn nhãn needs-human.
/rejectTừ chối — dừng, không code.
/abortHuỷ job hiện tại.
10
Định tuyến lệnh · hệ thống
Webhook issue_comment → server phân loại → worker xử lý: /approve đẩy job phase2; /revise tăng đếm & phân tích lại; /reject|/abort dừng.
Pha 2 — Code + Test → hoàn thiện PR

Chỉ chạy sau /approve. Đầu ra: PR có commit code + test, chuyển từ nháp sang sẵn-sàng-review.

11-12
/auto-implement · Claude (tự động)
Worker checkout lại branch của PR → claude -p /auto-implement: sửa source thật, viết test, chạy test, git commit + git push origin HEAD lên đúng branch PR.
13-14
Un-draft + ghi memory · hệ thống
Gọi GraphQL markPullRequestReadyForReview (PR hết nháp). Ghi bài học vào memory. Nếu ảnh hưởng FE → tự mở job impact cho repo FE kèm api_contract. Dọn checkout /tmp.
Bạn làm: review PR (giờ đã có code + test) như review của một dev bình thường → merge.

4. Cheat sheet

Tag trong commit (trigger Pha 1)

[zdc:update-be PRD-005]   # backend
[zdc:update-fe PRD-005]   # frontend
[zdc:update-qa PRD-005]   # qa

Điều kiện: nhánh ≠ main/master/develop · commit có tag · có đổi file po/**.

Comment trên PR (cổng người)

/approve
/revise cần thêm xử lý lỗi 409
/reject
/abort

Lệnh phải đứng đầu comment. /revise tối đa 3 lần.

5. Bằng chứng đã chạy thật

PR #4 — "Impact analysis: PRD-005 → be" chạy tự động trọn vẹn qua webhook thật:
  • Push po/PRD-005 + tag → server phân loại impact → worker tạo PR nháp #4.
  • Comment /approve → Pha 2 → Claude sửa src/index.js (+12/-2), viết list-orders.test.js (+35), commit feat(orders): add GET /orders [PRD-005], push.
  • finalizeMR → PR #4 draft = false. Tổng 3 commit, 4 file. Không can thiệp tay.

6. Vận hành & an toàn

Cơ chếMô tả
DRY_RUN=1Chạy thử: phân tích nhưng không tạo PR / không push. Đặt 0 để chạy thật.
HARNESS_PAUSED=1Kill-switch: worker bỏ qua mọi job mới (tạm dừng hệ thống).
Checkout tạmClone nằm trên VPS tại /tmp/zdc-checkout-*, tự xoá sau mỗi job (gồm sau approve). Code thật luôn ở GitHub.
MemoryBài học/issue lưu ở /data/memory.sqlite (volume) — nạp lại cho lần phân tích sau.
Số vòng reviseTối đa 3; quá thì gắn nhãn needs-human để người vào tay.

7. Câu hỏi mở

  • Webhook đang dùng HTTP (domain sslip.io) — production nên gắn domain + TLS.
  • Worker concurrency = 1 (xử lý tuần tự). Khi tải tăng cần xem lại scaling + cô lập checkout song song.
  • Phần FE handoff (auto mở job cho repo FE) đã có khung nhưng chưa demo end-to-end.
