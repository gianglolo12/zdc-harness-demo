ZDC Harness — Cách trigger workflow

Tài liệu cho team. Harness tự động hoá: PRD → phân tích tác động → bạn duyệt → code + test → Pull Request. Bạn chỉ cần "ra lệnh", harness lo phần còn lại và mở PR trên repo code.

TL;DR — 4 bước
  1. PO/BA: mở 1 GitHub Issue trên repo control-plane, tiêu đề [zdc:update-be <PRD-ID>].
  2. Harness comment "đã nhận" rồi mở draft PR (kèm phân tích tác động) trên repo BE.
  3. Dev: đọc PR, comment /approve → harness code + test + đẩy commit, bỏ nháp.
  4. Dev: review + merge PR như bình thường → Issue tự đóng.

Cách 1 — GitHub Issue ⭐ khuyên dùng

Hợp PO · không cần git · không sửa file
  1. Vào repo control-plane → tab Issues → New issue.
  2. Tiêu đề đặt đúng cú pháp tag: 
    [zdc:update-be G3-F07]
  3. Body ghi gì cũng được. Bấm Submit.
  4. ~30s sau, Issue có comment: "🤖 Đã nhận — đang phân tích G3-F07…"
  5. Vài phút sau, draft PR hiện trên repo source (tiêu đề Impact analysis: G3-F07 → be).

Cách 2 — Push (khi đang sửa chính nội dung PRD)

Hợp dev/BA đang chỉnh PRD
  1. Trên control-plane, tạo feature branch (KHÔNG phải main).
  2. Sửa file dưới po/**, commit message chứa tag: 
    git commit -m "docs(po): cập nhật G3-F07 [zdc:update-be G3-F07]"
git push origin <feature-branch>
  3. Phần còn lại giống Cách 1.
Lưu ý: push thẳng vào main không trigger (tránh kích hoạt khi merge). Phải là feature branch.

Cú pháp tag

[zdc:update-<role> <PRD-ID>]
  • <role>: be (backend), fe, qa — hiện đã wire be.
  • <PRD-ID>: id feature, dùng gạch nối (không dấu chấm): G3-F07, G4-F11.
[zdc:update-be G3-F07]   → BE làm feature Tạo đơn hàng
[zdc:update-be G4-F11]   → BE làm feature Tra cứu thẻ

Cổng duyệt — comment trên PR

Sau khi draft PR mở, dev điều khiển bằng cách comment lên PR đó:

CommentTác dụng
/approveĐồng ý → harness code + test + đẩy commit, bỏ nháp PR
/revise <góp ý>Phân tích lại kèm góp ý (tối đa 3 lần)
/rejectTừ chối — dừng, không code
/abortHuỷ job

Comment thường (không có lệnh) → bị bỏ qua.

Vòng đời đầy đủ (ví dụ G3-F07)

PO: New Issue "[zdc:update-be G3-F07]" │ ▼ Harness: comment "🤖 Đã nhận…" → mở draft PR #N trên zdc-be-demo │ (phân tích: đọc PRD + FRS + api-contract) ▼ Dev: đọc PR #N → comment "/approve" │ ▼ Harness: viết code Spring Boot + mvn test + push → PR hết nháp │ ▼ Dev: review PR #N → Merge │ ▼ Harness: đóng Issue + comment evidence trên PR

Câu hỏi thường gặp

  • PR mở ở đâu? Trên repo source (BE), không phải control-plane.
  • Bao lâu? Phân tích ~2-4 phút; code (sau approve) vài phút–chục phút tuỳ độ lớn feature.
  • Agent đọc gì? PRD (po/) và tài liệu kỹ thuật BE (be/<PRD>/: FRS + api-contract.yaml + diagrams) — kết hợp cả hai.
  • Đổi convention/rule cho BE? Sửa bundle be/ trong control-plane (be/rules/*, be/CLAUDE.md, be/commands/*) rồi push — hiệu lực ngay job kế, không cần deploy lại harness.
  • Feature quá lớn? Nên chia nhỏ PRD/feature khi dispatch để PR dễ review.

Lưu ý vận hành (người quản trị)

  • Webhook trên cả 2 repo: control-plane (push, issues) và source (issue_comment, pull_request).
  • DRY_RUN=1: chỉ phân tích, không tạo PR/không push (chế độ thử).
  • HARNESS_PAUSED=1: tạm dừng toàn bộ (kill-switch).
